# Code Review: Save/Load Persistence System

## ✅ ISSUE FIXED: Race Condition in Hydration

### The Problem
When user reloaded the page after saving, the layout reset to default instead of loading the saved version. Root cause: **The `_hydrated` flag was being persisted to localStorage**, creating a timing bug.

**Timeline of the bug:**
1. User saves layout → localStorage contains `{ cityLayout: [...], _hydrated: true }`
2. Page reloads → Zustand loads persisted state synchronously
3. State now has `_hydrated: true` BEFORE `onRehydrateStorage` callback runs
4. PhaserGame sees `hydrated=true` and immediately creates scene
5. But `cityLayout` is still the default because callback hasn't run yet!
6. Result: Scene renders with default layout while callback is still loading the saved version

### The Fix
**Don't persist the `_hydrated` flag.** Only persist the actual data. The flag is now:
- Initially `false` at startup
- Set to `true` ONLY in `onRehydrateStorage` callback AFTER data is loaded
- This ensures CityScene always waits for actual data

## 🔍 CODE REVIEW: Each Function

### 1. `useCityStore` — Persist Middleware Configuration

**Before (BROKEN):**
```javascript
partialize: (state) => ({ 
  districts, 
  cityLayout, 
  layoutVersion, 
  _hydrated: true   // ❌ RACE CONDITION
})
```

**After (FIXED):**
```javascript
partialize: (state) => ({
  districts,
  cityLayout,
  layoutVersion
  // ✅ _hydrated NOT persisted - only set in callback
})
```

**Why this matters:** The flag is runtime state, not user data. It should only be set AFTER hydration completes.

### 2. `onRehydrateStorage` Callback

**Issues Found:**
- ❌ Unclear flow with verbose logging
- ❌ Didn't explain WHY version check exists
- ❌ Inconsistent logging format

**Improved:**
```javascript
onRehydrateStorage: () => (state) => {
  console.group('🔄 [REHYDRATE] Hydrating from localStorage')
  
  // Clear intent: validate version
  if (state && state.layoutVersion !== LAYOUT_VERSION) {
    // Reset if version changed (protects against schema changes)
    state.cityLayout = defaultCityLayout
    state.layoutVersion = LAYOUT_VERSION
  } else if (state?.cityLayout?.length > 0) {
    console.log('✅ Loaded', state.cityLayout.length, 'objects')
  }
  
  // ✅ Set flag AFTER data is loaded
  state._hydrated = true
  console.groupEnd()
}
```

**Clarity improvements:**
- Clear guard clauses
- Explicit "mark hydration complete" step
- Early exit paths obvious

### 3. `updateCityLayout` Function

**Issues Found:**
- ❌ Confusing comment "bypass Zustand middleware" (not accurate)
- ❌ Redundant variable assignment
- ✅ Logic is correct but could be clearer

**Improved:**
```javascript
updateCityLayout: (newLayout) => {
  // ✅ Update store first for immediate UI feedback
  set({ cityLayout: newLayout })
  
  // ✅ Save to localStorage immediately (don't rely on middleware timing)
  const saveData = {
    districts: get().districts,
    cityLayout: newLayout,
    layoutVersion: LAYOUT_VERSION
  }
  localStorage.setItem('life-city-os-save', JSON.stringify(saveData))
}
```

**Why:** Direct localStorage writes avoid timing issues with Zustand's async persisting. This runs on every drag-drop.

### 4. `saveLayout` Action

**Before (OVER-ENGINEERED):**
```javascript
saveLayout: () => {
  const state = get()
  console.log('💾 [saveLayout] Manual save triggered')
  console.log('   Current cityLayout has', state.cityLayout.length, 'objects')
  
  localStorage.setItem('life-city-os-save', JSON.stringify({...}))
  
  console.log('✅ [saveLayout] Layout saved to localStorage')
  
  // Verify write (overkill - always succeeds)
  const saved = localStorage.getItem('life-city-os-save')
  if (saved) {
    const parsed = JSON.parse(saved)
    console.log('   Verified: localStorage has', parsed.cityLayout.length, 'objects')
  }

```

**After (CLEAN):**
```javascript
saveLayout: () => {
  const state = get()
  const saveData = {
    districts: state.districts,
    cityLayout: state.cityLayout,
    layoutVersion: LAYOUT_VERSION
  }
  localStorage.setItem('life-city-os-save', JSON.stringify(saveData))
  console.log(`💾 [saveLayout] Saved ${state.cityLayout.length} objects`)
}
```

**Why:**
- No verification needed (localStorage write is synchronous and reliable)
- Concise logging
- Same behavior as `updateCityLayout` (consistency)

### 5. `PhaserGame` — Hydration Polling

**Issues Found:**
- ❌ Confusing logic flow (if/else with same effect)
- ❌ No timeout (could hang forever)
- ❌ Missing early return in success path

**Improved:**
```javascript
useEffect(() => {
  // Fast path: already hydrated
  if (hydrated) {
    console.log('✅ [PhaserGame] Hydration complete')
    setIsHydrated(true)
    return  // ✅ Early return
  }

  // Slow path: wait for hydration (up to 2 seconds)
  console.log('⏳ [PhaserGame] Waiting for store hydration...')
  let attempts = 0
  const poll = setInterval(() => {
    attempts++
    const state = useCityStore.getState()
    
    if (state._hydrated) {
      console.log(`✅ [PhaserGame] Hydration complete (${attempts * 50}ms)`)
      setIsHydrated(true)
      clearInterval(poll)
    } else if (attempts > 40) {
      // Timeout safety: allow 2 seconds max
      console.warn('⚠️  [PhaserGame] Timeout, proceeding anyway')
      setIsHydrated(true)
      clearInterval(poll)
    }
  }, 50)

  return () => clearInterval(poll)
}, [hydrated])
```

**Key improvements:**
- ✅ Early return for clarity
- ✅ Timeout protection (fail-safe)
- ✅ Tracks elapsed time in logs
- ✅ Guards against infinite waits

## 📊 Data Flow (FIXED)

### On Page Load:
```
1. Page starts
2. Zustand initializes with defaults
3. Persist middleware loads from localStorage
   (only cityLayout, districts, layoutVersion — NOT _hydrated)
4. onRehydrateStorage callback runs
   - Validates version
   - Sets _hydrated = true
5. React re-renders with hydrated = true
6. PhaserGame sees hydrated = true
7. Creates Phaser scene
8. CityScene reads useCityStore.getState().cityLayout
   ✅ NOW contains saved layout (not default)
9. spawnBuildings() creates buildings from saved layout
```

**Key:** Step 8 now has correct data because we waited for hydration.

## 📝 Summary

**Root Cause:** Persisting the `_hydrated` flag created a race condition where PhaserGame started before data loaded.

**Solution:** 
1. Stop persisting `_hydrated` flag (runtime state, not data)
2. Set `_hydrated = true` ONLY in `onRehydrateStorage` callback
3. Add timeout safety in polling loop
4. Simplify logging for clarity

**Result:** Layouts now persist correctly across page reloads. ✅
